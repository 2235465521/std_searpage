## 2. 首次部署指南

### 2.1 拉取代码
```bash
cd /home/zkbz01/
git clone git@github.com:2235465521/std_searpage.git std_searpage_v2
cd std_searpage_v2
```

### 2.2 编译前端
要求服务器已安装 Node.js (推荐 18+)。
```bash
cd /home/zkbz01/std_searpage_v2/frontend
npm install
npm run build
```

### 2.3 启动后端服务 (Gunicorn)
要求服务器已安装 Miniconda/Anaconda，并已创建环境及安装依赖。
```bash
cd /home/zkbz01/std_searpage_v2/backend

# 激活虚拟环境
conda activate std_searpage

# 同步数据库并收集静态文件（视首次部署情况执行）
python manage.py migrate
python manage.py collectstatic --noinput

# 以守护进程(后台)模式启动后端 Gunicorn 服务
gunicorn --workers 3 --bind 127.0.0.1:8001 config.wsgi:application --daemon
```

### 2.4 配置 Nginx 代理服务器
项目根目录中已提供了一份标准的 `nginx.conf`，里面内置了前端静态路径、后端 API 反向代理以及文件下载提速的 X-Accel-Redirect 内部路由。
```bash
cd /home/zkbz01/std_searpage_v2/

# 将项目内的配置复制到 Nginx 系统目录
sudo cp nginx.conf /etc/nginx/sites-available/std_searpage.conf

# 如果是首次配置，需要建立软链接启用该配置（如果已存在可忽略）
sudo ln -s /etc/nginx/sites-available/std_searpage.conf /etc/nginx/sites-enabled/

# 检查 Nginx 语法并重载
sudo nginx -t
sudo nginx -s reload
```

## 3. Nginx 核心配置解析 (参考)

我们在项目根目录的 `nginx.conf` 中约定了以下核心规则：

```nginx
server {
    listen 8089;
    server_name 47.106.104.48; # 您的服务器外网 IP 或域名

    # 1. 前端静态文件托管
    root /home/zkbz01/std_searpage_v2/frontend/dist;
    index index.html;

    # 2. 前端 React Router 路由支持
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 3. 后端静态文件（Django Admin 等）
    location /static/ {
        alias /home/zkbz01/std_searpage_v2/backend/staticfiles/;
    }

    # 4. 【核心性能优化】标准文件的高速下载内部路由
    # Django 发送 X-Accel-Redirect 响应头后，Nginx 会接管该路径的物理文件分发
    location /protected-files/ {
        internal;
        alias /mnt/std_bk/磁盘阵列/标准文件处理/;
    }

    # 5. 后端 API 接口反向代理
    location /api/v1/ {
        proxy_pass http://127.0.0.1:8001/api/v1/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

## 4. 日常升级指令 (更新代码)

当代码在 GitHub 更新后，需要在服务器上执行升级。建议养成 **先拉代码 -> 编前端 -> 重启后端 -> (可选)重载 Nginx** 的规范流程。

### 第一步：拉取最新代码
```bash
cd /home/zkbz01/std_searpage_v2
# 如果本地有未提交的临时修改导致冲突，可执行 git checkout -- . 放弃本地修改
git pull
```

### 第二步：更新前端 (只有前端改动时必做)
```bash
cd /home/zkbz01/std_searpage_v2/frontend
npm run build
# 前端构建完成后立即生效，因为 Nginx 直接托管了 dist 目录
```

### 第三步：重启后端服务 (只有后端改动时必做)
后端 Python 驻留在内存中，代码修改后**必须重启**。
由于直接使用的 gunicorn 守护进程，我们采用“先杀旧，再起新”的安全策略：
```bash
# 1. 强制杀死正在占用 8001 端口的旧版本 Gunicorn 进程
sudo kill -9 $(sudo lsof -t -i:8001)

# 2. 重新进入后端目录并以守护进程模式启动新版代码
cd /home/zkbz01/std_searpage_v2/backend
conda activate std_searpage
gunicorn --workers 3 --bind 127.0.0.1:8001 config.wsgi:application --daemon
```

### 第四步：更新 Nginx (仅当 nginx.conf 被修改时才做)
```bash
cd /home/zkbz01/std_searpage_v2
sudo cp nginx.conf /etc/nginx/sites-available/std_searpage.conf
sudo nginx -s reload
```
