# 增加注册功能 - 后端架构设计与实施文档

## 1. 核心目标
增加人员注册接口，并严格遵循“权限隔离”与“职责分离”架构。**注册权限必须且仅能由超级管理员（Super Admin）拥有**。

## 2. 实施路径

### 2.1 新增超级管理员权限拦截器 (`permissions.py`)
由于 DRF 自带的 `IsAdminUser` 主要是判断 `is_staff`（是否能登录后台），为了最高级别的严格控制，我们将自定义 `IsSuperAdmin` 权限类。
- **职责**：在请求进入视图逻辑前，检查 Token 对应的用户是否登录，并且 `is_superuser == True`。若非超管，直接拦截并返回 403 Forbidden。

### 2.2 隔离业务服务层 (`services.py`)
- **增加函数**：`create_user_service(username, password, email, role)`。
- **职责**：
  - 校验用户名是否已存在。
  - 调用 `CustomUser.objects.create_user()`（由于 Django 会自动加盐 hash 密码，**严禁使用 `objects.create()` 明文存密码**）。
  - 根据 `role` 入参判断是否赋予 `is_superuser = True` 或 `is_staff = True`。
  - 只负责数据库原子操作，不接触 Request 对象。

### 2.3 新增视图控制层 (`views.py`)
- **增加类**：`UserRegisterView(APIView)`
- **职责**：
  - 配置 `permission_classes = [IsSuperAdmin]` 挂载超管屏障。
  - 提取 Request 中的 `username`, `password`, `email`, `role`。
  - 做基本非空校验后，将数据抛给 `create_user_service`。
  - 捕获可能的 ValueError，组装并返回 `{ "code": 0, "message": "注册成功", "data": ... }` 格式的标准 JSON。

### 2.4 路由发布 (`urls.py`)
- 挂载路由：`path('auth/register', UserRegisterView.as_view(), name='auth_register')`
